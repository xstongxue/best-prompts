# 写作质量标准

## 内容层面（必须满足）

1. **背景说"为什么"**：「背景与目标」要说清楚这件事的出发点和目的，不只是介绍做了什么
   - 🚫 差：`本月完成了模型测试工作`
   - ✅ 好：`为评估不同硬件配置下 Qwen3.5 系列模型的实际可用性，支撑产品部署方案决策，本月在 NVIDIA 和华为 NPU 两套硬件上完成了 11 款模型的系统测试`

2. **成果要量化**：优先给出可量化的成果指标，定性亮点作为补充
   - 🚫 差：`模型测试效果不错`
   - ✅ 好：`Qwen3.5-122B-A10B-AWQ 完成 18 题模拟卷用时 30 分钟，错题率 1/18，成为 NVIDIA 4×24G 方案最优选择`

3. **不足要挖根因**：「不足与改进」必须分析根本原因，改进方向要具体可执行
   - 🚫 差：`数学公式查重不准，需要加强`
   - ✅ 好：`数学公式查重准确率低（根因：公式经 tokenization 后短 token 分词排序与文本不一致），改进方向：针对公式类型引入结构化比对逻辑，跳过纯 token 序列匹配`

4. **经验要可复用**：「经验与启示」提炼的是方法论，而非流水账，对后续工作有实际指导价值
   - 🚫 差：`要多沟通，加强测试`
   - ✅ 好：`MOE 架构模型在 NPU 上部署时，建议优先测试 multistream_overlap_shared_expert 等参数，可显著降低重复性思考失败率`

## 结构层面

- 「成果与亮点」和「不足与改进」并重，避免只写成绩、回避问题
- 「不足与改进」用表格，包含"根因分析"列，深度区别于简单罗列
- 「经验与启示」是总结的点睛之笔，提炼方法论，不重复前面内容

## 语言层面

- 诚实具体，不美化、不掩盖问题
- 专业术语、数据原样保留
- 不写"加强沟通""积极探索"等无实质内容的表述
